== Introduction

SUSE Observability, anciennement connu sous le nom de StackState, peut être utilisé pour l'observabilité de vos clusters Kubernetes et de leurs charges de travail.

L'installation de SUSE Observability, de l'extension SUSE Observability UI et des agents SUSE Observability prend environ 30 minutes au total.

== Obtenir de l'aide

Pour obtenir de l'aide, veuillez déposer un dossier d'assistance sur https://scc.suse.com/[Customer Center (SCC)] SUSE.

== Conditions préalables

=== Clé de licence

Une clé de licence pour le serveur SUSE Observability peut être obtenue via le SUSE Customer Center dans l'onglet Abonnement et sera affichée en tant que code d'enregistrement "SUSE Observability". Cette licence est valable jusqu'à la fin de votre abonnement à Rancher Prime.

=== Exigences

Pour installer SUSE Observability, assurez-vous que le cluster dispose d'une capacité de CPU et de mémoire suffisante. Vous trouverez ci-dessous les exigences spécifiques.

Il existe différentes options d'installation pour {stackstate-product-name}. Il est possible d'installer {stackstate-product-name} dans une configuration à haute disponibilité (HA) ou à instance unique (non-HA). La configuration non-HA est recommandée à des fins de test ou pour des environnements restreints. Pour les environnements de production, il est recommandé d'installer {stackstate-product-name} dans une configuration HA.

La configuration de production HA peut prendre en charge de 150 à 4000 nœuds observés. Dans ce tableau de dimensionnement, un nœud observé correspond à 4 vCPU et 16 Go de mémoire, soit notre site `default node size`.
Si les nœuds de votre cluster observé sont plus grands, ils peuvent compter pour plusieurs `default nodes`, ainsi un nœud de 12vCPU et 48GB compte pour 3 `default nodes` sous observation lors de la sélection d'un profil.
L'installation Non-HA peut prendre en charge jusqu'à 100 `default nodes` sous observation.

=== Stockage

SUSE Observability utilise des réclamations de volumes persistants pour les services qui doivent stocker des données. La classe de stockage par défaut du cluster sera utilisée pour tous les services à moins qu'elle ne soit remplacée par des valeurs spécifiées sur la ligne de commande ou dans un fichier `values.yaml`. Tous les services sont fournis avec une taille de volume préconfigurée qui devrait vous permettre de démarrer, mais qui peut être personnalisée ultérieurement à l'aide de variables, si nécessaire.

[NOTE]
====
SUSE Observability exige que le stockage sous-jacent soit basé sur la mémoire flash (SSD) ou sur des performances similaires.
====
[NOTE]
====
Pour les environnements de production, NFS n'est pas recommandé ni pris en charge pour le provisionnement du stockage dans SUSE Observability en raison du risque potentiel de corruption des données.
====

=== Les différents composants

==== SUSE Observability Server

Il s'agit de la partie de l'installation concernant le serveur hébergé sur site. Il contient un ensemble de services permettant de stocker les données d'observabilité :

* Topologie (StackGraph)
* Métriques (VictoriaMetrics)
* Traces (ClickHouse)
* Journaux (ElasticSearch)

En outre, il contient un ensemble de services pour toutes les tâches d'observabilité, par exemple les notifications, la gestion de l'état, la surveillance, etc.

==== SUSE Observability Agent

L'agent léger SUSE Observability est installé sur vos noeuds de travail en aval. Il recueille et rapporte des mesures, des événements, des traces et des journaux, et il fournit une observabilité et des informations en temps réel, permettant une surveillance et un dépannage proactifs de votre environnement informatique.

La version SUSE Observability de l'agent utilise également eBPF comme moyen léger de surveiller tous vos environnements système et leurs communications. Il décode également les signaux RED (Rate, Errors and Duration) pour la plupart des protocoles L7 courants tels que TCP, HTTP, TLS, Redis, etc.

==== Rancher Prime - Extension de l'interface utilisateur pour l'observabilité

Il s'agit d'une extension de l'interface utilisateur de Rancher Manager qui intègre les signaux de santé observés par SUSE Observability. Il donne un accès direct à l'état de santé de n'importe quelle ressource et un lien vers l'interface utilisateur de SUSE Observability pour un examen plus approfondi.

=== Où installer le serveur SUSE Observability

Le serveur SUSE Observability doit être installé dans son propre cluster en aval prévu pour Observability. Voir l'image ci-dessous à titre de référence.

Pour que SUSE Observability puisse fonctionner correctement, il faut que les conditions suivantes soient réunies

* le cluster d'observabilité pour prendre en charge un moyen d'exposer SUSE Observability sur une URL HTTPS à Rancher, aux utilisateurs de SUSE Observability et à l'agent SUSE Observability. Cela peut se faire via une configuration Ingress utilisant un contrôleur Ingress, ou bien un loadbalancer (cloud) pour les services SUSE Observability. Pour plus d'informations, consultez la https://ranchermanager.docs.rancher.com/how-to-guides/new-user-guides/kubernetes-resources-setup/load-balancer-and-ingress-controller[documentation de] Rancher.

=== Pré-installation

Avant d'installer le serveur SUSE Observability, une classe de stockage par défaut doit être configurée dans le cluster où le serveur SUSE Observability sera installé :

* *Pour les k3 :* La classe de stockage local-path de type rancher.io/local-path est créée par défaut.
* *Pour EKS, AKS, GKE*, une classe de stockage est définie par défaut.
* *Pour les pilotes de nœuds RKE2*: Aucune classe de stockage n'est créée par défaut. Vous devrez en créer un avant d'installer SUSE Observability.

=== Installation

. Obtenir le plan de barre
+
.helm_repo.sh
[,text]
----
helm repo add suse-observability https://charts.rancher.com/server-charts/prime/suse-observability
helm repo update
----
+
. Commande permettant de générer des fichiers de valeurs de diagramme de barre :
+
.helm_template.sh
[,text]
----
export VALUES_DIR=.
helm template \
  --set license='<your license>' \
  --set baseUrl='<suse-observability-base-url>' \
  --set rancherUrl='<rancher-prime-base-url>' \
  --set sizing.profile='<sizing.profile>' \
  suse-observability-values \
  suse-observability/suse-observability-values --output-dir $VALUES_DIR
----

L'adresse `baseUrl` doit être l'URL par laquelle SUSE Observability sera accessible à Rancher, aux utilisateurs et à l'agent SUSE Observability. Elle doit correspondre à l'URL qu'un utilisateur saisira dans le navigateur pour ouvrir l'interface utilisateur. Elle doit donc inclure le schéma et tout port personnalisé, par exemple `https://observability.internal.mycompany.com` (qui utilise le port par défaut). Voir aussi l <<_accessing_suse_observability,'accès à SUSE Observability>>.

Le site `sizing.profile` devrait être un essai, 10-nonha, 20-nonha, 50-nonha, 100-nonha, 150-ha, 250-ha, 500-ha. Sur la base de ces profils, le fichier `sizing_values.yaml` est généré et contient des tailles par défaut pour les ressources et la configuration de SUSE Observability à déployer sur un mode Ha ou NonHa. Par exemple 10-nonha produira un `sizing_values.yaml` destiné à déployer une instance NonHa de SUSE Observability pour observer un cluster de 10 noeuds en mode Non High Available. Actuellement, il n'est pas possible de passer d'un environnement non ha à un environnement ha. Par conséquent, si vous pensez que votre environnement nécessitera l'observation d'environ 150 nœuds, il est préférable d'opter immédiatement pour un environnement ha.

Pour voir les informations de santé dans Rancher à l'aide de l'extension de l'interface utilisateur, définissez la valeur `rancherUrl` à l'URL de Rancher (pour être précis, son Origine).

Cette commande génère les fichiers `$VALUES_DIR/suse-observability-values/templates/baseConfig_values.yaml`, `$VALUES_DIR/suse-observability-values/templates/sizing_values.yaml`, et `$VALUES_DIR/suse-observability-values/templates/affinity_values.yaml` contenant la configuration nécessaire à l'installation de SUSE Observability Helm Chart.

[NOTE]
====
Le mot de passe de l'administrateur de SUSE Observability sera autogénéré par la commande ci-dessus et sera affiché sous forme de commentaires dans le fichier `basicConfig.yaml` généré. Pour plus d'informations, voir xref:/setup/security/authentication/single_password.adoc[mot de passe unique].
Les valeurs réelles contiennent les hachages `bcrypt` de ces mots de passe afin qu'ils soient stockés en toute sécurité dans la version Helm du cluster.
====


[warn]
====
L'utilisation d'un mot de passe unique par défaut est idéale pour démarrer avec SUSE Observability, mais pour une configuration de production, xref:/setup/security/authentication/authentication_options.adoc[des options d'authentification plus sûres] sont disponibles.
====


[NOTE]
====
Conservez les fichiers générés `basicConfig.yaml`, `sizing_values.yaml` et `affinity_values.yaml` en toute sécurité. Vous pouvez réutiliser ces fichiers pour les mises à niveau, ce qui permet de gagner du temps et de garantir que SUSE Observability continue d'utiliser la même clé API. Ceci est souhaitable car cela signifie que les agents et autres fournisseurs de données pour SUSE Observability n'auront pas besoin d'être mis à jour.
Les fichiers peuvent être régénérés indépendamment en utilisant les commutateurs `basicConfig.generate=false` et `sizing.generate=false` pour désactiver l'un d'entre eux tout en conservant la version précédemment générée du fichier dans le fichier `output-dir`.
====

[NOTE]
====
Le tableau SUSE Observability Values génère des configurations d'affinité que vous pouvez utiliser avec le tableau SUSE Observability principal pour contrôler le comportement de l'ordonnancement des pods. Reportez-vous à xref:/setup/install-stackstate/kubernetes_openshift/affinity.adoc[Configurer les affinités Kubernetes]) pour plus d'informations.
====

== Privilèges requis

Le déploiement de l'agent SUSE Observability nécessite les privilèges système suivants :

. `hostPID: true`: Ce privilège est nécessaire pour associer les identificateurs de processus (PID) aux groupes de contrôle (cgroups) correspondants. Cette association est essentielle pour établir une correspondance précise entre les processus et leurs conteneurs respectifs.
. `hostNetwork: true` (facultatif) : Par défaut, l'agent de nœud s'exécute avec `hostNetwork: true` pour faciliter la récupération des données de métriques ouvertes de tous les pods configurés sur le nœud sans nécessiter de politiques de réseau supplémentaires. S'il est désactivé, des règles de réseau appropriées doivent être définies pour que l'agent puisse accéder aux points d'extrémité nécessaires.
. `securityContext.privileged: true`: Ce privilège élevé est nécessaire pour plusieurs fonctions critiques. Il permet principalement à l'agent d'injecter des programmes eBPF (extended Berkeley Packet Filter) dans chaque espace de noms du réseau à des fins de surveillance. Il est également nécessaire pour lire les tables de suivi des connexions (conntrack) dans tous les espaces de noms du réseau. Bien que cette liste ne soit pas exhaustive, les développements futurs visent à remplacer ce large privilège par des capacités Linux plus granulaires lorsque cela est possible.

En outre, l'agent a besoin que les sockets d'exécution des conteneurs soient montés dans son pod. Cette configuration est essentielle car elle facilite la communication directe avec les démons d'exécution des conteneurs, ce qui est une condition préalable à l'extraction des métriques et des métadonnées de tous les conteneurs sur le système hôte.

== Modèle de message d'intérêt public restreint aux éleveurs

La configuration `Rancher-restricted` (link:https://ranchermanager.docs.rancher.com/how-to-guides/new-user-guides/authentication-permissions-and-global-configuration/psa-config-templates[Pod Security Admission (PSA) Configuration Templates]) est une configuration très restrictive qui correspond aux meilleures pratiques actuelles en matière de sécurisation des pods.

Lorsque vous exécutez Rancher sur un cluster Kubernetes qui applique une politique de sécurité restrictive par défaut, il existe deux façons d'installer la carte SUSE Observability Helm :

* *Exempter l'ensemble de l'espace de noms du graphique*, ainsi que d'autres link:https://ranchermanager.docs.rancher.com/how-to-guides/new-user-guides/authentication-permissions-and-global-configuration/psa-config-templates#exempting-required-rancher-namespaces[espaces de noms Rancher requis par].
* *Désactiver le conteneur privilégié Elasticsearch init* en définissant `elasticsearch.sysctlInitContainer.enabled` à `false`. Pour ce faire, vous devez augmenter manuellement les link:https://www.elastic.co/docs/deploy-manage/deploy/self-managed/vm-max-map-count[paramètres de mémoire virtuelle de] (`vm.max_map_count`) sur les nœuds. Voir aussi xref:/setup/install-stackstate/kubernetes_openshift/required_permissions.adoc#_elasticsearch[Elasticsearch Required Permissions]. 

Étant donné que SUSE Observability Agent doit fonctionner en mode privilégié, il est recommandé de l'installer dans un espace de noms que vous prévoyez d'exempter de la politique restrictive.

[NOTE]
====
Tous les conteneurs SUSE Observability Helm chart sont configurés avec les paramètres `securityContext` suivants à partir de la version `v2.3.8`:

. `securityContext.capabilities.drop` est `["ALL"]`
. `securityContext.seccompProfile.type` est `"RuntimeDefault"`
. `securityContext.runAsNonRoot` est `true`
. `securityContext.allowPrivilegeEscalation` est `false`
====
